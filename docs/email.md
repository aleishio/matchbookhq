# Email Delivery

YC OS sends application email through Resend from a dedicated sending subdomain.
Do not verify or send from the root domain. A subdomain keeps app email
reputation separate from normal company mail and avoids MX conflicts.

## Resend Setup

1. In Resend, add a domain that includes the sending subdomain, for example:

```text
events.matchbookhq.com
```

2. Copy the DNS records generated by Resend into the DNS provider. Resend shows
   SPF, DKIM, and DMARC records under the domain's Records tab. If receiving or
   return-path records are enabled, copy those generated records too.
3. Wait for Resend to mark the subdomain verified.
4. Create a sending API key scoped to this domain when possible.
5. Store the real values in Vercel Environment Variables and local secret files,
   never in this repo.

Official references:

- https://resend.com/docs/send-with-nextjs
- https://resend.com/docs/dashboard/domains/introduction
- https://resend.com/docs/dashboard/domains/tracking

## Environment Variables

```text
RESEND_API_KEY=re_...
RESEND_SENDING_DOMAIN=events.matchbookhq.com
RESEND_FROM_EMAIL=yc@events.matchbookhq.com
RESEND_FROM_NAME=YC OS
RESEND_REPLY_TO_EMAIL=yc@matchbookhq.com
```

`RESEND_FROM_EMAIL` must use `RESEND_SENDING_DOMAIN` exactly. For example,
`yc@events.matchbookhq.com` is valid when `RESEND_SENDING_DOMAIN` is
`events.matchbookhq.com`; `yc@matchbookhq.com` is rejected as a sender because
it uses the root domain.

`RESEND_REPLY_TO_EMAIL` can point at the root-domain mailbox. For Matchbook,
set it to `yc@matchbookhq.com` so replies go to the existing inbox while
delivery still comes from the `events.matchbookhq.com` sending subdomain.

The installed Resend SDK requires Node 20 or newer. Use Node 20+ locally and
in deployment.

## Code Boundary

Use the server-only helper:

```ts
import { sendEmail } from "@/lib/email/resend";

await sendEmail({
  to: "founder@example.com",
  subject: "Intro draft",
  text: "Hello from YC OS",
});
```

Do not import `@/lib/email/resend` from client components. It reads
`RESEND_API_KEY` and constructs the provider client.

## Replies and Tracking

The future backend should use Resend Receiving plus Webhooks, not Resend
Automations, to ingest founder replies and link/open events. The proposed
Supabase schema and backend phases are documented in
`docs/email-replies-schema.md`.

Current Resend-side configuration:

- Receiving is enabled for `events.matchbookhq.com`.
- Receiving MX is verified.
- Open and click tracking are enabled.
- Tracking subdomain is `links.events.matchbookhq.com`.
- Webhook endpoint is `https://matchbookhq.com/api/resend/webhook`.

Pending DNS record:

```text
CNAME  links.events     links1.resend-dns.com
```

## Event Approval Clarification Emails

The approvals workflow uses email when an applicant cannot be confidently
matched to YC records from the Lu.ma application alone.

User-triggered flows:

- `needs_info` applicants: ask for YC company, batch, and best YC-connected
  email.
- `manual` applicants: ask for the claimed YC relationship or investor/network
  context.
- Bulk send: selected rows in the current queue/filter can receive clarification
  emails together, but each email still needs its own durable send job and audit
  row.
- The send-info action starts from the default subject/body, but an ops user can
  edit the outbound subject and message before queueing. The exact user-authored
  copy is stored on the email job payload for later reply review.

Backend requirements:

- Send only from a verified subdomain sender.
- Keep the root-domain mailbox available as reply-to if needed.
- For approval clarification emails that should be parsed by YC OS, use a
  receiving-enabled Resend address on the verified subdomain so replies reach
  `POST /api/resend/webhook`.
- Store the outbound email intent before calling Resend.
- Store the exact subject/body that was queued, including user-authored edits.
- Store the Resend email ID after send succeeds.
- Link each outbound email to the event application and approval decision that
  triggered it.
- Leave the Lu.ma application pending while waiting for a reply.
- Move the application to `awaiting_reply` after the email is queued/sent.
- Drain queued clarification emails with `POST /api/resend/clarification-emails`
  using the server action secret. This route sends through Resend, sets
  `replyTo` to the job's receiving-enabled sender, records `resend_email_id`,
  and retries failures with backoff.

The UI should show the email as queued/sent, not as proof of identity.

## Reply Analysis For Approvals

Inbound replies should be stored first and analyzed second. The AI analysis can
recommend `ready` or `manual`, but irreversible Lu.ma writebacks still require a
user-triggered approval or rejection unless a future policy explicitly allows a
specific auto-approval rule.

Approval reply analysis should extract:

- YC company name.
- YC batch.
- YC-connected email candidate.
- Applicant role: founder, investor, guest, employee, or unknown.
- Relationship explanation.
- Evidence summary.
- Recommended queue: `ready`, `manual`, `needs_info`, or `rejected_candidate`.
- Human-readable reason.

If the reply does not prove a mapped YC identity or trusted network
relationship, keep the application in manual review.

Implemented reply ingestion:

- `POST /api/resend/webhook` verifies Resend Standard Webhook headers
  (`svix-id`, `svix-timestamp`, `svix-signature`) with
  `RESEND_WEBHOOK_SECRET`.
- Every verified webhook is stored in `provider_webhook_events` with provider
  `resend` and `svix-id` as the provider event ID.
- For `email.received`, the route uses `RESEND_API_KEY` to fetch the full
  received email content from Resend Receiving, then links the reply to the
  latest clarification job by applicant email.
- Linked replies are inserted into `applicant_replies`; parsed fields are
  advisory and Lu.ma writebacks remain user-triggered.
- Resend Automations are not part of inbound reply capture. They can be added
  later only after YC OS emits its own post-analysis event.

## Operational Routes

Both routes are server-only:

- `POST /api/resend/clarification-emails`: protected by
  `LUMA_SYNC_SECRET`, `CRON_SECRET`, or `WEBHOOK_SECRET`; claims queued
  `clarification_email_jobs` and sends them through Resend.
- `POST /api/resend/webhook`: public Resend webhook endpoint protected by
  `RESEND_WEBHOOK_SECRET` signature verification.
